# FE-040 — MSW handlers + per-sub-router fixtures + GitHub Pages deploy switch

> Depends on: All Phase 0–7 views complete (mock fixtures must mirror the
> shapes those views actually consume).
> Followed by: FE-040a (retrospective UI round-trip across Phase 4–7
> against the fixtures shipped here).

## Why this ticket exists

1. The frontend has no mock layer — `view/src/api/mock/` does not exist.
2. `npm run build:mock` script is wired in `package.json` but only sets an
   env var; with no MSW handlers behind the flag it is a no-op.
3. `.github/workflows/deploy.yml` currently runs `npm run build` (not
   `build:mock`), so every push to `main` deploys a build with no
   backend behind it — every `/api/*` call 404s on GitHub Pages. The
   live demo has been broken since the route went up.
4. Phases 4 / 6 / 7 each ran with an empty dev DB and could not
   exercise the full CRUD / chart / dashboard paths end-to-end. Mock
   fixtures double as deterministic UI test data, so building them now
   unblocks the retrospective round-trips in FE-040a.

This ticket installs MSW, ships realistic per-sub-router fixtures,
wires the worker into the SPA bootstrap, and flips the GitHub Pages
deploy job to use `npm run build:mock`.

## Notes

- MSW v2 surface: import handlers from `msw` (`http`, `HttpResponse`),
  worker from `msw/browser`. Do not import from `msw/node` — that is the
  Node test runner.
- The committed `mockServiceWorker.js` in `public/` is generated by the
  MSW CLI; never hand-edit it. Re-run `npx msw init public/ --save`
  after upgrading the `msw` package.
- All response bodies must use the `ApiResponse` envelope (`{status: 1,
  data, msg: "success"}`) so the existing `request.ts` interceptor
  unwraps them correctly. The bug fixed in FE-001 (`res.msg` vs
  `res.message`) means handlers must include `msg` even on errors.
- "In-memory CRUD" means each handler module owns a `let` array, and
  POST/PUT/DELETE mutate that array. This way the mock site behaves like
  a real backend within a session — refresh resets state. Do not try to
  persist across reloads.
- Fixture realism budget: aim for ≥ 3 rows per resource and at least one
  branch per StatusBadge variant. Do not over-engineer — the goal is
  exercising every UI code path, not modelling a real user's data.
- `vite.config.ts` `base: '/networth/'` means the worker registration
  URL is `/networth/mockServiceWorker.js`, not `/mockServiceWorker.js`.
  Use `import.meta.env.BASE_URL + 'mockServiceWorker.js'` to keep dev
  (base `/`) and prod (base `/networth/`) in sync.
